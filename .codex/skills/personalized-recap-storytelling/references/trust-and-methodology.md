# Trust And Methodology

Every beat needs its own trust contract.

## Per-Beat Contract

For each story beat, define:
- `Metric basis`: what actually determines the result
- `Time window`: what period counts
- `Filters`: what is excluded
- `Reference class`: compared to whom
- `Tie-breaks`: what happens when close values compete
- `Eligibility`: minimum data needed
- `User controls`: which settings or privacy controls affect the result

## Principles

- Choose the metric that best fits the story, not one universal metric for every beat.
- Surprising is good only when it is still defensible.
- A recap is a reflection, not a forensic ledger of every event.
- If the methodology is curated, say so internally and design accordingly.

## Disputed-Result Pattern

If users challenge a result, have a standard recovery:
1. explain what counted
2. explain what did not count
3. explain why this ranking method was chosen
4. acknowledge that memory is imperfect
5. explain what the user can change for future recaps

## Reference-Class Pitfalls

Comparisons are meaningless without context.
Always state whether the beat is relative to:
- the user’s own history
- a cohort
- a club or role
- all users
- invited friends in a shared mode

## AI Or Generative Storytelling

If generation is used, keep a creative contract:
- every claim must be traceable to real behavior
- no hallucinated specifics
- no stereotyping
- no overconfident emotional claims

Evaluate generated outputs for:
- accuracy
- tone
- safety
- formatting
