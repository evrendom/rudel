# Eligibility And Fallbacks

Recaps fail when every user is forced through the same run.

## Define Eligibility At Three Levels

- `Recap-level`: is there enough data to show any recap at all?
- `Chapter-level`: is there enough data to enter a subsection?
- `Beat-level`: is this specific card justified?

## Minimum Requirements

For every beat, define:
- data threshold
- confidence threshold
- tie-break rule
- fallback copy or alternate beat
- route-out if it should not render

## Sparse-Data Rules

- Never shame low-activity users.
- Shorten the run instead of padding it with weak cards.
- Replace “top” language with “so far” or “early signal” language when needed.
- Use broad reflective beats before advanced identity beats.

## Zero-Data Route

A zero-data user should not hit a broken recap.
Provide one of:
- a no-data summary with clear reasons
- a milestone explainer
- a future-facing setup flow

## Good Fallback Behavior

- Replace missing precision with broader framing.
- Swap weak social comparison for self-reflection.
- Route from a failed deep beat to a stronger broad beat.

## Bad Fallback Behavior

- Fake numbers
- empty placeholders
- vague “you are unique” language
- exposing internal null states to the user
